# 실시간 설비 모니터링 대시보드 — 하네스(Harness) 실습

> 한국디스플레이(가상)의 200대 생산설비가 쏟아내는 **이기종 실시간 텔레메트리**를 해석·정규화하여
> 공정관리 모니터링 대시보드로 만드는 [revfactory/harness](https://github.com/revfactory/harness) 실습 교안입니다.
>
> **완성 데모**: https://namojo.github.io/dashboard-display/ · **소스**: [namojo/dashboard-display](https://github.com/namojo/dashboard-display)

## 1. 문제 정의

디스플레이 FAB에는 Canon·Nikon 노광기, TEL 코터, 국산 식각·세정 설비, 물류 자동화 설비 등 수백 대가 실시간으로 생산정보를 송신합니다. 문제는 **설비마다 말하는 언어가 다르다**는 것입니다.

| 벤더 계열 | 페이로드 특징 | 예시 |
|---|---|---|
| Canon (SECS/GEM풍) | 숫자 상태코드, 축약 키 | `{OpState: 1, GlassCnt: 2728, Yld: 0.9782}` |
| Nikon·TEL·AMAT | snake_case, **ms 단위**, 세미콜론 알람 문자열 | `{run_mode:"PRODUCTIVE", cycle_time_ms: 83000}` |
| 국산 (원익·세메스 등) | **한글 키** | `{상태:"가동", 생산수량: 2728, 수율: 97.82}` |
| 레거시 물류 | CSV 압축 문자열 | `"PHO-L1-01,FPA-6300,R,2728,…"` |

같은 "가동 중, 수율 97.8%"라는 사실이 4가지 형식·3가지 단위계·2가지 언어로 도착합니다. 대시보드가 어려운 이유는 차트가 아니라 **이 해석(정규화) 계층**입니다 — `xlsx/` 실습의 분임조 은어 문제가 실시간 스트림으로 확장된 셈입니다.

## 2. 실습 목표 (2단계 산출물)

1. **가상 실시간 API**: 200대 설비(9개 공정 × SEMI E10 상태머신)의 텔레메트리를 실제처럼 생성 — 벤더별 이기종 페이로드, 상태 전이, 알람 발생, 지표 랜덤워크 포함
2. **관제 대시보드**: API를 소비하여 OEE·가동률·UPH KPI, 200타일 설비 그리드, 실시간 알람 피드, 설비 상세(원시 vs 표준 페이로드 대비)를 표시

> GitHub Pages는 정적 호스팅이므로 API는 브라우저 내에서 REST 동작(지연 30~120ms, 간헐 503, 쿼리 필터)을 충실히 모사합니다. 대시보드는 `Api.get('/api/v1/…')` 계약만 바라보므로 실서버 전환 시 이 한 계층만 교체하면 됩니다.

## 3. 하네스 구성

이 도메인은 **계약(스키마)을 사이에 둔 생산자-소비자 구조**라서, 파이프라인이 아닌 **계약 중심 팬아웃** 패턴이 적합합니다. 표준 스키마(SET v1)를 먼저 확정하면 시뮬레이터와 대시보드를 병렬 개발할 수 있습니다.

```
                ┌──────────────────┐
                │ schema-architect │  ← 최우선: SET v1 표준 스키마 + E10 상태 사전 확정
                └────────┬─────────┘
          ┌──────────────┼──────────────┐   (스키마 확정 후 병렬 팬아웃)
          ▼              ▼              ▼
 [sim-api-builder] [normalizer-eng] [dashboard-builder]
  설비 시뮬레이터      벤더 어댑터 구현      관제 UI 구현
          └──────────────┼──────────────┘
                         ▼
                  [qa-validator]     ← 경계면 교차 비교 (incremental)
```

| 에이전트 | 핵심 역할 | 전용 스킬 (예) |
|---|---|---|
| `schema-architect` | 표준 스키마 SET v1, SEMI E10 상태 매핑 사전, API 계약(엔드포인트·쿼리) 정의 | — (산출물이 곧 계약) |
| `sim-api-builder` | 200대 설비 상태머신 + **벤더별 원시 페이로드 생성** + REST 모사(지연·오류 주입) | `equipment-telemetry` — 공정·벤더 카탈로그, 알람 코드북, 현실적 지표 범위 |
| `normalizer-eng` | 4계열 어댑터 구현. 단위 변환(ms→s, 비율→%), 미지 상태값은 `[미확인]` 격리 | `vendor-payload-dict` — 벤더별 필드 매핑 사전 (선언적, 코드와 분리) |
| `dashboard-builder` | KPI·그리드·알람·상세 UI. 503 재시도, 로딩·오류 상태 처리 | `frontend-design` — 관제실 다크 테마 토큰 |
| `qa-validator` | ① 시뮬레이터 원시값 ↔ 정규화 결과 ↔ 화면 표시값 3점 교차 비교 ② 필터·상태전이 시나리오 테스트 | `webapp-testing` — Playwright 스모크 |

### 하네스 설계 포인트

- **schema-architect가 병렬화의 열쇠**: 스키마 없이 시작하면 시뮬레이터와 대시보드가 서로 다른 필드명을 가정하고 통합 단계에서 붕괴합니다. 계약 확정 → 팬아웃이 순서입니다.
- **매핑 사전은 스킬로 외부화**: `vendor-payload-dict` 스킬에 "Canon의 `Yld`는 0~1 비율이므로 ×100" 같은 규칙을 선언적으로 담으면, 신규 벤더 추가 시 코드가 아닌 사전에 행만 추가합니다 — `xlsx/` 표준용어집과 동일 원리.
- **QA는 3점 교차 비교**: "화면에 숫자가 보인다"가 아니라, 같은 설비의 (원시 `GlassCnt`) = (정규화 `produced`) = (화면 표시값)을 **동시에 읽어 비교**해야 정규화 버그를 잡습니다. 실제 본 실습 구현 중에도 "라인 필터 시 타일이 안 숨겨지는" 버그를 Playwright 교차 검증이 잡아냈습니다.

## 4. 실행 프롬프트

```
/harness 한국디스플레이의 200대 생산설비(노광·증착·식각·세정·코터현상·
AOI검사·봉지·본딩·물류, 3개 라인) 실시간 모니터링 시스템을 만들고자 합니다.

[1단계 — 계약 확정]
표준 설비 텔레메트리 스키마(SET v1)와 SEMI E10 상태 매핑 사전,
REST API 계약(/api/v1/equipments, /kpi/summary, /alarms)을 먼저 정의하세요.

[2단계 — 병렬 구현]
- 가상 API: 설비별 상태머신(RUN/IDLE/DOWN/PM/SETUP 전이 확률),
  벤더별 이기종 페이로드(Canon: SECS 숫자코드 / Nikon·TEL: snake_case·ms단위 /
  국산: 한글 키 / 레거시: CSV 문자열), 알람 발생, 지연·간헐 503을 실제처럼 모사
- 정규화 계층: 4계열 → SET v1 어댑터. 매핑 규칙은 선언적 사전으로 분리하고,
  사전에 없는 상태값·필드는 임의 해석하지 말고 [미확인]으로 격리 보고
- 대시보드: OEE·가동률·합산UPH·누적생산·알람 KPI, 공정 섹션별 설비 그리드
  (상태 색상), 실시간 알람 피드, 라인 필터, 설비 상세(원시 vs 표준 페이로드
  나란히 표시 — 정규화 가치를 시연하는 핵심 화면)

[3단계 — 검증]
같은 설비에 대해 원시값 = 정규화값 = 화면값 3점 교차 비교를 수행하고,
라인 필터·상태 전이·503 재시도 시나리오를 Playwright로 테스트하세요.

[불변 규칙]
- 대시보드는 Api 계층의 계약만 바라본다 (시뮬레이터 내부 직접 접근 금지)
- 단위는 표준 스키마에서 통일한다 (초, %, 장) — 변환은 어댑터의 책임
- 정적 배포(GitHub Pages) 가능해야 하므로 빌드 도구 없는 순수 HTML/JS로
```

## 5. 검수 체크리스트

- [ ] 설비 200대가 모두 그리드에 표시되고, 상태 색상(SEMI E10 5종)이 실시간 전이하는가
- [ ] 상세 패널에서 Canon 설비는 `OpState: 1`, 국산 설비는 `상태: "가동"` 등 **서로 다른 원시 형식**이 보이고, 표준 스키마는 동일한가
- [ ] 단위 변환 정확성: Nikon `cycle_time_ms: 83000` → 표준 `tactSec: 83`, Canon `Yld: 0.978` → `yieldPct: 97.8`
- [ ] 3점 교차: 원시 `GlassCnt` = 정규화 `produced` = 화면 생산수량
- [ ] 라인 필터 시 해당 라인 설비만 표시되는가 (전체 200 → L2 약 67)
- [ ] 간헐 503에서 대시보드가 죽지 않고 재시도로 복구하는가
- [ ] 콘솔 에러 0건 (Playwright 기준)

## 6. 교육 포인트

**① 어려운 것은 차트가 아니라 해석이다.** 대시보드 실패의 대부분은 시각화가 아니라 데이터 정규화에서 옵니다. 이 실습은 그 지점을 정면으로 다룹니다 — 상세 패널에서 원시/표준 페이로드를 나란히 보여주는 화면이 곧 "정규화 계층의 가치 증명"입니다.

**② 계약이 병렬성을 만든다.** 스키마(SET v1)를 먼저 확정했기에 시뮬레이터·정규화·대시보드를 팬아웃으로 병렬 개발할 수 있습니다. 에이전트 팀에서 계약 없는 병렬화는 통합 단계의 재작업으로 되돌아옵니다.

**③ 시뮬레이터 우선 개발(Sim-first).** 실설비를 연결하기 전에 "실제처럼 나쁜"(이기종, 지연, 간헐 오류) 가상 API를 먼저 만들면, 대시보드의 견고함(재시도, 로딩·오류 상태)을 안전하게 훈련할 수 있습니다. 시뮬레이터는 테스트 픽스처이자 교보재입니다.

**④ 매핑 사전의 자산화.** 벤더 필드 매핑은 코드에 흩뿌리지 않고 선언적 사전으로 모읍니다. 신규 설비 도입 = 사전에 어댑터 1개 추가. 분임조 표준용어집(xlsx) → 정책 용어(slide) → 식당 명칭 통합(map) → 설비 페이로드(dashboard)로 이어지는, 이 교육과정 전체를 관통하는 단일 원리입니다.

## 7. 심화 과제

1. **신규 벤더 온보딩 훈련**: 5번째 페이로드 형식(예: XML 문자열)을 시뮬레이터에 추가하고, `vendor-payload-dict` 사전만 수정해 대시보드 코드 변경 없이 수용해 보세요.
2. **이상 탐지 에이전트**: UPH 급락·수율 이탈을 감지해 알람 피드에 `[예측]` 태그로 주입하는 분석 에이전트를 팀에 추가해 보세요.
3. **실서버 전환**: `api.js`를 FastAPI + SSE 실서버로 교체하고, 대시보드 코드가 계약 덕분에 무변경으로 동작하는지 확인해 보세요.
4. **일일 리포트 연계**: `slide/` 실습의 KRDS 파이프라인으로 "일일 설비 가동 보고" 슬라이드를 자동 생성하는 에이전트를 연결해 보세요.
