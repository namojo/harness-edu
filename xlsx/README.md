# 제조라인 공정 보고서 표준화 — AI 에이전트 실습

> **비정형 현장 문서 → 표준화된 통합 데이터**로 변환하는 에이전트 워크플로우 실습 자료입니다.
> Claude Cowork, Claude Code, Codex 등 파일을 읽고 쓸 수 있는 AI 에이전트 환경에서 실행합니다.

## 1. 문제 정의

제조 공장의 분임조별로 공정관리 보고서를 엑셀로 작성해 보고합니다. 보고서의 **양식(구조)은 동일**하지만, 분임조마다 사용하는 **용어·은어·문체·표기 형식이 제각각**이어서 통합 관리가 어렵습니다.

이는 실제 제조 현장에서 흔한 문제입니다. 사람이 수작업으로 취합하면 매일 반복되는 단순 노동이 되고, 전통적인 코드(정규식·룩업 테이블)로 자동화하기엔 표현의 변형이 너무 다양합니다. **LLM 에이전트가 가장 잘하는 영역** — 의미 기반 정규화 — 이 필요한 지점입니다.

## 2. 시나리오

1. 삼성전자 냉장고 제조공장(가상)의 6개 분임조가 각자의 언어로 일일 생산보고서를 작성합니다.
2. 팀장은 이 6개 보고서를 **공정 표준용어**로 일치시키고, **동일한 표현 양식과 표준 작성 방식**으로 재작성한 통합 보고서를 원합니다.

## 3. 파일 구성

| 파일 | 역할 |
|---|---|
| `생산보고서_한마음조_A라인.xlsx` ~ `생산보고서_미래조_F라인.xlsx` | **입력**: 6개 분임조의 일일 생산보고서 (용어·문체 제각각) |
| `공정_표준용어집.xlsx` | **지식 베이스**: 용어 매핑 + 표준 양식 정의 (시트 3개) |
| `prompt.md` | **에이전트 지시서**: 통합 작업 프롬프트 |
| `통합_생산보고서.xlsx` | **산출물** (실습 후 에이전트가 생성) |

### 6개 분임조의 언어 스타일

각 보고서는 6개 공정 × (계획/생산/양품/불량/불량률/가동/비가동/비고) 구조는 같지만, 언어가 전부 다릅니다.

| 분임조 | 라인 | 언어 특성 | 예시 표현 |
|---|---|---|---|
| 한마음조 | A | 표준에 가까운 정석체 | 캐비닛 프레스, 불량, 비가동 |
| 불꽃조 | B | 현장 은어 + 일본어 잔재 | 빵꾸, 데나오시, 가타, 시마이 검사, 빠킹 |
| 새벽조 | C | 영어 혼용 | NG, Comp' Ass'y, L/S, Down Time |
| 장인조 | D | 축약 한자어 행정체 | 냉시조립, 검포, 적요, 특기사항 무 |
| 으뜸조 | E | 구어체 | "폼 쏘는 데", "라인 선 시간", "불량난거" |
| 미래조 | F | KPI 약어 스타일 | ACT, Loss율, U/T, D/T, Die C/O |

날짜 표기(`2026-07-02` / `26.7.2` / `20260702`), 합계 표기(합계/토탈/계/TTL), 문체까지 다르게 설계되어 있습니다.

### 표준용어집 구조 (`공정_표준용어집.xlsx`)

| 시트 | 내용 |
|---|---|
| `공정용어` | 6개 표준 공정명 + 공정코드(P10~P60) + 은어 매핑 |
| `관리항목용어` | 계획수량·불량률·재작업 등 18개 항목의 동의어 매핑과 표기 규칙 |
| `표준양식정의` | 컬럼 순서, 날짜 형식, 문체 규칙, 은어 처리 원칙 등 12개 작성 규칙 |

핵심 설계: 매핑 규칙을 프롬프트에 하드코딩하지 않고 **엑셀 파일로 외부화**했습니다. 새 은어가 발견되면 프롬프트 수정 없이 용어집에 행만 추가하면 됩니다. 현장 반장이 직접 유지보수할 수 있는 구조입니다.

## 4. 실습 방법

### A. Claude Cowork (권장 — 노코드)

1. 이 디렉토리의 엑셀 파일 7개(보고서 6개 + 용어집)를 작업 폴더에 넣습니다.
2. `prompt.md` 내용을 그대로 붙여넣어 실행합니다.
3. 에이전트가 파일을 읽고 → 용어를 정규화하고 → `통합_생산보고서.xlsx`를 생성하는 과정을 관찰합니다.

### B. Claude Code / Codex

이 저장소를 클론한 뒤 `xlsx` 디렉토리에서 Claude Code 또는 Codex를 실행하고, **`prompt.md`의 내용을 그대로 입력**하면 됩니다.

에이전트는 pandas/openpyxl 등으로 파일을 파싱하고, 용어집 기반 정규화 후 수식이 살아있는 통합 엑셀을 생성해야 합니다.

## 5. 산출물 요구사항 (검수 기준)

`통합_생산보고서.xlsx` — 시트 3개:

1. **라인별 표준보고** — 6개 라인 × 6개 공정을 표준 양식으로 재작성. 합계·불량률은 하드코딩이 아닌 **엑셀 수식** 사용
2. **전사 집계** — 공정코드별 크로스탭(공정 × 라인), 불량률 상위 공정 하이라이트
3. **이슈 요약** — 특이사항을 표준용어로 통합, `[조치요청]` / `[데이터 이상]` / `[미확인 용어]` 분류, 반복 이슈(예: 자재 결품) 집계

### 검수 체크리스트

- [ ] 6개 공정명이 모두 표준용어 + 공정코드(P10~P60)로 통일되었는가
- [ ] 은어(빵꾸, 데나오시, 콤프 등)가 표준용어로 치환되었는가 — 정보 손실 우려 시 `(원문: ○○)` 병기
- [ ] **수치가 원본과 100% 일치**하는가 (표현만 바꾸고 숫자는 불변)
- [ ] 양품 + 불량 = 생산수량 크로스체크를 수행했는가
- [ ] 합계·불량률이 수식으로 계산되는가 (`#REF!` 등 수식 오류 0건)
- [ ] 치환 이력(어느 조의 어떤 표현을 어떻게 바꿨는지)이 보고되었는가

## 6. 교육 포인트 — 왜 이렇게 설계했는가

이 실습은 단순 "엑셀 정리"가 아니라, 자연어로 명세한 **ETL 파이프라인(Extract → Transform → Load)**입니다. 에이전트 설계 관점에서 세 가지를 관찰하세요.

**① 지식의 외부화 (Knowledge as Data)**
치환 규칙이 프롬프트가 아닌 용어집 파일에 있습니다. 프롬프트는 "절차"만, 용어집은 "지식"만 담당합니다. RAG에서 추론과 지식을 분리하는 것과 같은 원리로, 지식이 바뀌어도 에이전트 로직은 그대로입니다.

**② "모르면 멈춰라" 규칙 (Uncertainty Escalation)**
용어집에 없는 표현은 임의 해석하지 않고 `[미확인 용어]`로 에스컬레이션합니다. LLM의 환각으로 데이터가 오염되는 것을 막는 안전장치입니다. 자율성이 높은 에이전트일수록 이 "불확실성의 명시적 출구"가 중요합니다.

**③ 검증과 감사 추적 (Verification & Audit Trail)**
수치 불변 제약, 크로스체크, 치환 이력 보고는 변환 결과를 사람이 신뢰·검수할 수 있게 하는 장치입니다. 제조 데이터처럼 수치가 의사결정에 직결되는 도메인에서는 필수입니다.

## 7. 심화 과제

1. **일일 배치화**: 이 프롬프트를 Claude Code Skill(SKILL.md + 용어집 동봉)로 패키징하여 "폴더에 파일 넣고 실행"하는 자동화 파이프라인으로 발전시켜 보세요.
2. **용어집 자가 성장**: `[미확인 용어]`로 보고된 표현을 사람이 승인하면 용어집에 자동 추가되는 Human-in-the-loop 루프를 설계해 보세요.
3. **이상 탐지 추가**: 불량률이 전일 대비 급증한 공정을 자동 플래그하는 분석 단계를 프롬프트에 추가해 보세요.
4. **도구 비교**: 같은 prompt.md를 Cowork / Claude Code / Codex에서 각각 실행하고 결과 품질·과정을 비교해 보세요.
