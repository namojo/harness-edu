# 단일 프롬프트 vs 하네스 — 과업지시서로 검증하는 A/B 실험

> 나라장터 과업지시서 1건(국가통합바이오빅데이터 EMR 연계 ETL 구축, 양산부산대학교병원)을
> 입력으로, **① 단일 프롬프트**와 **② revfactory/harness 전문가 에이전트 팀**으로 각각
> 사업 제안서(docx)를 작성하고 결과를 비교하는 실험 교안입니다.
> 두 방식의 실제 산출물과, 하네스가 생성하는 구성물(CLAUDE.md·에이전트·스킬) 전부가
> 이 디렉토리에 들어 있습니다.

## 1. 실험 질문

**"배경지식이 없는 상태에서, 전문가 에이전트 구조는 무엇을 바꾸는가?"**

과업지시서에는 FHIR, KCD-8, 심평원 EDI 같은 단어가 이름만 등장합니다. 발주 도메인
(보건의료 데이터)을 모르는 상태에서 좋은 제안서를 쓸 수 있는가 — 이것이 실험의
핵심 조건입니다.

## 2. 디렉토리 구성

```
harness/
├── README.md                        ← 본 교안
├── 과업지시서_데이터뱅크.hwpx        ← 입력 (나라장터 원문)
├── 제안서A_단일패스.docx             ← 트랙 A 산출물 (3쪽)
├── 제안서B_하네스.docx               ← 트랙 B 산출물 (8쪽)
├── artifacts/
│   └── 요구사항_인벤토리.md          ← 트랙 B의 중간 산출물 (차이의 근원)
├── CLAUDE.md                        ← 하네스: 프로젝트 불변 규칙
└── .claude/
    ├── agents/                      ← 하네스: 전문가 에이전트 5종
    │   ├── rfp-analyst.md           (요구사항 전수 추출)
    │   ├── domain-expert.md         (보건의료 표준 지식 주입)
    │   ├── solution-architect.md    (기술 제안 설계)
    │   ├── pm-expert.md             (WBS·리스크·조건 대응)
    │   └── compliance-reviewer.md   (전수성·확약 건전성 검증)
    └── skills/                      ← 하네스: 스킬 3종
        ├── rfp-analysis/            (RFP 전수 추출 방법론)
        ├── medical-data-standards/  (도메인 지식 사전)
        └── proposal-writing/        (제안서 구조·문체 표준)
```

## 3. 트랙 A — 단일 프롬프트 실행

Claude(또는 Claude Code)에 과업지시서를 첨부하고 한 번에 요청합니다.

```
첨부한 과업지시서를 기반으로 사업 제안서를 docx로 작성해줘.
```

`제안서A_단일패스.docx`가 그 결과입니다. **일부러 못 쓴 것이 아닙니다** — 구조가
잡혀 있고 문장도 매끄러운, 단일 패스의 "괜찮은 평균"입니다. 이 기준선이 정직해야
비교가 의미를 갖습니다.

## 4. 트랙 B — 하네스 실행

이 디렉토리를 Claude Code로 열면 `CLAUDE.md`, `.claude/agents/`, `.claude/skills/`가
자동 인식됩니다. 실행:

```
/harness 첨부한 과업지시서를 기반으로 사업 제안서 docx를 작성하고자 합니다.
```

또는 이미 구성된 팀을 직접 가동:

```
rfp-analyst → domain-expert → (solution-architect ∥ pm-expert) → 통합
→ compliance-reviewer 파이프라인으로 제안서를 작성하세요.
compliance-reviewer가 전건 통과를 승인할 때까지 수정 루프를 돌리세요.
```

### 파이프라인이 실제로 하는 일

| 단계 | 에이전트 | 산출물 | 단일 프롬프트에는 없는 것 |
|---|---|---|---|
| 1 | rfp-analyst | `artifacts/요구사항_인벤토리.md` — REQ 11 / DOM 13 / CON 7 / DEL 4 전수 추출·ID 부여 | 목적 문단에 숨은 "가명화", 기대효과에 숨은 "보안 연계망", 표 각주의 강행 규정을 REQ로 승격 |
| 2 | domain-expert | 표준코드별 매핑 전략 노트 | FHIR KR Core ValueSet, KCD-7→8 전환표, 원내코드↔EDI, 개인정보보호법 §28-2 — **문서 밖 지식** |
| 3 | solution-architect ∥ pm-expert | 기술 제안 / WBS·리스크·조건 대응 (병렬) | 주 단위 WBS·임계경로, "미매핑률 ≤1%" 같은 검증 가능한 확약 |
| 4 | compliance-reviewer | 컴플라이언스 검증표 | 인벤토리 ID 전건을 원고에서 대조 — 누락 발견 시 담당자 지목 수정 |

## 5. 결과 비교 — 차이가 물성으로 남는 4개 지점

`제안서A_단일패스.docx`와 `제안서B_하네스.docx`를 나란히 열고 확인하세요.

| 비교 지점 | A (단일 패스) | B (하네스) |
|---|---|---|
| **배경지식** | "표준코드에 맞추어 매핑합니다" — 과업지시서 단어의 반복 | 6종 코드체계별 매핑 전략 표: KR Core ValueSet, KCD 전환표, 원내-EDI 매핑, ATC 2단계 (3.2절) |
| **전수 대응** | 눈에 띄는 요구 위주 — 보안 연계망·비정형 가명화 취급이 얕음 | REQ-01~11 추적 매트릭스, 검수 기준표로 재사용 명시 (2장) |
| **조건 조항** | 사실상 언급 없음 | CON-01~07 전건 수용 확약표: 3년 유지보수 SLA, 라이선스 검토서, 상주 5인, HW/SW 부담 (5장) |
| **검증 가능성** | "1단계 약 3주", "품질을 확보" | W1~W9 WBS + 임계경로, 미매핑률 ≤1%·가명화 누출 0건 + 이행 방안 병기 |

한 줄 요약: **A는 과업지시서를 요약한 제안서, B는 과업지시서에 응답한 제안서.**

## 6. 교육 포인트 — 차이는 어디서 왔는가

**① 차이의 근원은 문장이 아니라 중간 산출물.** B의 모든 우위는
`artifacts/요구사항_인벤토리.md` 하나에서 파생됩니다. 전수 추출·ID 부여라는
지루한 작업이 추적 매트릭스, 조건 확약표, 검수 기준표를 낳습니다. 하네스란
이 중간 산출물의 생산을 역할로 강제하는 장치입니다.

**② "배경지식 없음"의 해법은 지식의 사전화.** domain-expert가 똑똑해서가 아니라,
`medical-data-standards` 스킬에 지식이 사전으로 축적돼 있어서 가능합니다.
다음 의료 RFP에서는 이 사전이 그대로 재사용됩니다 — 이 과정 전체를 관통하는
"매핑 사전의 자산화" 원리의 제안서 버전입니다.

**③ 검증자 분리가 누락을 잡는다.** 조건 조항 7건은 잘 쓰려는 노력으로 잡히지
않습니다. "인벤토리 ID를 하나씩 원고에서 찾아라"라는 별도 역할(compliance-reviewer)의
기계적 대조가 잡습니다. 잘 쓰는 것과 빠뜨리지 않는 것은 다른 능력입니다.

**④ A가 이기는 경우도 안다.** 내부 검토용 초안, 참고용 요약이라면 A의 속도가
정답입니다. 하네스는 "탈락 사유가 조항 하나의 누락일 수 있는" 고부담 문서에서
정당화됩니다 — 비용(과정의 복잡함)과 리스크(누락의 대가)를 저울질하는 것이
이 실험의 최종 교훈입니다.

## 7. 심화 과제

1. **평가위원 롤플레이**: 별도 세션에서 Claude에게 조달 평가위원 역할을 주고
   A·B를 채점시켜 보세요(요구 이해도/기술/관리/조건 대응 4개 항목). 점수 차이가
   어느 항목에서 벌어지는지 확인하세요.
2. **인벤토리 파괴 실험**: 요구사항 인벤토리에서 CON 섹션을 삭제하고 트랙 B를
   재실행해 보세요 — compliance-reviewer가 원문 대조로 이를 잡아내는지(에이전트
   정의의 에러 핸들링 조항)가 하네스 견고성의 시험대입니다.
3. **타 도메인 이식**: 다른 분야 과업지시서(예: 교육·물류)를 넣고, 무엇이 그대로
   동작하고(rfp-analysis, proposal-writing) 무엇을 교체해야 하는지
   (medical-data-standards → 해당 도메인 사전) 확인하세요. 스킬 분리 기준이
   왜 그렇게 설계됐는지 체감됩니다.
4. **codex/ 트랙 연결**: 이 에이전트 5종을 `codex/` 실습의 TOML 형식으로 번역해
   Codex 서브에이전트로 같은 실험을 수행해 보세요.
