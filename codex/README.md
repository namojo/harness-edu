# Codex로 배우는 세 가지 일하는 방식 — 트렌드 분석 보고서 실습

> 한국디스플레이(가상) **전략마케팅팀**이 "디스플레이 × AI 최신 트렌드 분석 보고서"를 만듭니다.
> 같은 보고서를 세 가지 방식으로 만들어 보며, OpenAI Codex의
> **단일 에이전트 → 멀티 에이전트(서브에이전트) → 하네스형 시스템**을 단계적으로 익힙니다.
> 코딩 지식이 없어도 따라올 수 있습니다 — Codex는 코드만이 아니라 조사·분석·문서 작업의 도구입니다.

## 0. 한눈에 보는 세 단계

| 단계 | 비유 | 방식 | 어울리는 상황 |
|---|---|---|---|
| **Level 1** | 유능한 인턴 한 명 | 프롬프트 하나로 처음부터 끝까지 | 오늘 오후 임원 미팅 전 브리핑이 필요할 때 |
| **Level 2** | 분야별 전문가 팀 | 서브에이전트 병렬 분업 + 팀장 취합 | 분기 심층 보고서 — 깊이와 폭이 둘 다 필요할 때 |
| **Level 3** | 보고서 생산 **시스템** | 역할·규칙·템플릿을 파일로 자산화 | 매달 반복되는 정례 보고 — 누가 돌려도 같은 품질 |

핵심 질문은 "어느 게 제일 좋은가"가 아니라 **"이 일은 몇 단계짜리 일인가"**입니다.

## 1. 준비

```bash
npm install -g @openai/codex   # 또는 brew install codex
codex login
git clone https://github.com/namojo/harness-edu.git
cd harness-edu/codex
```

이 디렉토리에는 Level 2·3에서 쓸 예시 파일이 미리 들어 있습니다.

```
codex/
├── README.md                        ← 본 교안
├── AGENTS.md                        ← 팀의 보고서 작성 원칙 (Codex가 자동으로 읽음)
├── .codex/agents/                   ← 전문가 에이전트 정의 5종
│   ├── tech-scout.toml              (디스플레이 기술 동향)
│   ├── ai-analyst.toml              (AI × 디스플레이 접점)
│   ├── market-analyst.toml          (시장·경쟁사)
│   ├── report-writer.toml           (보고서 집필)
│   └── fact-checker.toml            (출처·수치 검증)
└── .agents/skills/trend-report/SKILL.md  ← 보고서 표준 템플릿·출처 규칙
```

## 2. Level 1 — 단일 에이전트: 인턴 한 명에게 맡기기

프롬프트 하나로 조사부터 집필까지 한 번에 시킵니다. `--search`를 붙이면 실시간 웹 검색을 사용합니다.

```bash
codex exec --search "당신은 한국디스플레이 전략마케팅팀의 리서처입니다.
'디스플레이 × AI 최신 트렌드 분석' 보고서를 report/trend-brief.md 로 작성하세요.

다뤄야 할 범위:
1) 디스플레이 기술: OLED·마이크로LED·QD, 최근 전시회(CES/SID) 발표
2) AI와의 접점: AI 글래스·XR 기기용 디스플레이, 온디바이스 AI 탑재
   기기의 패널 수요, AI 기반 공정·검사 혁신
3) 시장·경쟁: 주요 패널사(삼성D·LGD·BOE·CSOT) 동향과 수급 전망

규칙: 모든 수치·주장에 출처와 날짜를 붙이고, 마지막에
'전략적 시사점' 3가지와 '이번 분기 액션 제안'을 담으세요. 분량 2~3쪽."
```

10분 남짓이면 초안이 나옵니다. **관찰 포인트**: 빠르고 간편하지만, ① 세 분야를 한 명이 훑다 보니 깊이가 고르지 않고 ② 조사자와 검증자가 같은 사람이라 자기 글의 오류를 잘 못 봅니다. 이 한계가 Level 2의 존재 이유입니다.

---

## 3. Level 2 — 멀티 에이전트: 전문가 팀에게 나눠 맡기기

Codex는 요청하면 **서브에이전트**를 생성해 병렬로 일을 시킬 수 있습니다. `.codex/agents/`의 전문가 정의를 사용해, 대화형 세션(`codex`)에서 이렇게 지시합니다.

```
디스플레이 × AI 트렌드 분석 보고서를 만들려고 합니다.

[1단계] 다음 3개 에이전트를 병렬로 생성해 각자 담당 분야를 조사하고
research/ 폴더에 분야별 조사 노트를 쓰게 하세요. 셋 모두 끝날 때까지 기다리세요.
- tech-scout   → research/tech.md
- ai-analyst   → research/ai.md
- market-analyst → research/market.md

[2단계] report-writer 에이전트를 생성해 research/의 노트 3개를
trend-report 스킬의 표준 템플릿에 맞춰 report/trend-report.md 로 통합 집필하게 하세요.

[3단계] fact-checker 에이전트를 생성해 보고서의 모든 수치·출처를
원문과 대조 검증하고, 문제 항목을 [검증실패] 목록으로 보고하게 하세요.
검증 실패 항목은 report-writer 에게 수정시키세요.
```

**관찰 포인트**:
- 세 분야가 **동시에** 조사되므로 각각의 깊이가 확보됩니다.
- 집필자(report-writer)와 검증자(fact-checker)가 **다른 에이전트**입니다 — 자기 글을 자기가 검사할 때 생기는 확증 편향을 구조적으로 피합니다. 사람 조직의 교차 검토와 같은 원리입니다.
- 서브에이전트끼리는 직접 대화하지 않습니다. 부모(여러분과 대화 중인 Codex)가 산출물을 확인하고 다음 단계로 넘기는 **팀장 역할**을 합니다. 그래서 "research/에 뭘 어떤 형식으로 남길지"가 미리 합의돼 있어야 하죠 — 그 합의를 파일로 굳힌 것이 Level 3입니다.
- 비용 주의: 병렬 분업은 같은 일을 단일 에이전트로 할 때보다 토큰을 더 씁니다. 깊이가 그 비용을 정당화할 때 선택하세요.

---

## 4. Level 3 — 하네스형 시스템: 매달 돌아가는 보고서 공장

Level 2까지는 "이번 보고서"를 만드는 방법이었습니다. Level 3은 질문이 바뀝니다: **"다음 달에도, 담당자가 바뀌어도, 같은 품질이 나오게 하려면?"**

답은 프롬프트가 아니라 **파일**입니다. 이 디렉토리가 그 실물입니다.

| 파일 | 사람 조직에 비유하면 | 역할 |
|---|---|---|
| `AGENTS.md` | 팀 업무 수칙 게시판 | 출처 필수, 추측 금지 같은 불변 규칙. Codex가 시작할 때 자동으로 읽음 |
| `.codex/agents/*.toml` | 직무기술서 5장 | 누가(전문가) 무엇을 어떤 기준으로 하는지. 세션이 끝나도 남는 자산 |
| `.agents/skills/trend-report/` | 보고서 표준 양식 + 작성 매뉴얼 | 어떻게(템플릿·출처 표기·문체). 양식이 바뀌면 이 파일만 수정 |

이 구조 위에서는 매달 실행이 한 줄로 줄어듭니다.

```bash
codex exec --search "trend-report 스킬과 .codex/agents/ 의 전문가 팀으로
2026년 7월호 트렌드 보고서를 report/2026-07.md 로 생산하세요.
지난 호(report/2026-06.md)와 비교해 '전월 대비 변화' 섹션을 포함하세요."
```

더 나아가면 이 한 줄을 GitHub Actions 등에 넣어 **매월 1일 자동 실행**되는 정례 보고 파이프라인이 됩니다. 여기까지 오면 여러분이 만든 것은 보고서가 아니라 **보고서를 만드는 시스템**입니다 — `xlsx/`의 표준용어집, `map/`의 프로젝트 스펙과 같은 원리로, 지식과 절차를 프롬프트 밖으로 꺼내 파일로 자산화한 것입니다.

**관찰 포인트**: Level 1과 Level 3의 실행 명령은 둘 다 한 줄입니다. 차이는 명령의 길이가 아니라, **그 한 줄이 기대는 자산의 양**입니다.

## 5. 비교 실험 (수업 활동)

같은 주제로 세 레벨을 모두 실행하고 비교표를 채워 보세요.

| 지표 | L1 단일 | L2 팀 | L3 시스템 |
|---|---|---|---|
| 소요 시간 | | | |
| 분야별 깊이 (각 1~5점) | | | |
| 출처 누락·오류 수 (fact-check 기준) | | | |
| 토큰/비용 | | | |
| "다음 달에도 같은 품질" 가능성 | | | |

예상되는 발견: L1은 속도, L2는 깊이와 검증, L3은 재현성에서 이깁니다. 그리고 L3의 진짜 비용은 실행이 아니라 **초기 설계**(직무기술서와 템플릿 작성)에 있다는 것 — 조직 만들기가 원래 그렇듯이요.

## 6. 교육 포인트

**① 에이전트 수는 품질이 아니라 구조의 문제.** 급한 브리핑에 5명을 투입하는 건 낭비고, 정례 보고를 매번 즉흥 프롬프트로 하는 건 도박입니다. 일의 성격(일회성/심층/반복)이 단계를 결정합니다.

**② 분업의 전제는 합의된 양식.** Level 2에서 서브에이전트들은 서로 대화하지 못하므로, research/ 노트의 형식과 보고서 템플릿이 곧 팀의 공용어입니다. 사람 팀도 마찬가지죠 — 이 과정 전체를 관통하는 "계약이 병렬성을 만든다"의 마케팅 버전입니다.

**③ 검증자는 반드시 분리.** 조사·집필·검증을 한 에이전트에 몰아주면 오류가 그대로 통과합니다. fact-checker를 별도 에이전트로 두는 것은 기술 선택이 아니라 품질 설계입니다.

**④ 반복되는 일은 프롬프트가 아니라 파일로.** 세 번 이상 반복할 일이라면 AGENTS.md·에이전트 정의·스킬로 굳히세요. 이것이 "하네스형으로 일한다"의 전부입니다 — Claude Code든 Codex든.

## 7. 심화 과제

1. **전문가 증원**: `policy-analyst.toml`(각국 디스플레이·AI 정책/보조금 동향)을 직접 작성해 팀에 추가하고, 보고서에 4번째 섹션이 생기는지 확인해 보세요.
2. **템플릿 교체 실험**: 스킬의 템플릿을 "임원용 1쪽 요약" 버전으로 바꾸고 재실행 — 에이전트 정의는 그대로인데 산출물 형식만 바뀌는 것(누가/어떻게의 분리)을 확인하세요.
3. **크로스 플랫폼 검증**: `.agents/skills/trend-report`를 Claude Code의 `.claude/skills/`로 복사해 `/harness`로 같은 팀을 꾸려 보세요. 어느 부분이 그대로 이식되고 어느 부분이 번역이 필요한지 기록하세요.
4. **자동화**: Level 3 실행 명령을 GitHub Actions cron에 등록해 "매월 1일 오전 9시 자동 발행"을 완성해 보세요.
