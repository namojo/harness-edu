# 트랙 B — 업무 자동화: 코드 없이 반복 업무를 시스템으로

> 코드를 짜지 않는 실무자를 위한 트랙입니다. 금융회사 **준법·리스크팀**이 매주 반복하는
> "규제·공시 모니터링 리포트"를 소재로, Codex를 **정례 업무 자동화 엔진**으로 쓰는 법을
> 배웁니다. 배우는 무기는 세 가지 — 라이브 검색(`--search`), 출력 계약(`--output-schema`),
> 규칙 파일(AGENTS.md). 셋을 합치면 "매주 누가 돌려도 같은 형식"의 리포트 공장이 됩니다.

트랙 A가 "코드를 고친다"였다면, 트랙 B는 "코드를 안 짜도 반복 업무를 파일로 굳힌다"입니다.
같은 5원칙(상위 README) 위에서 도메인만 바뀝니다.

## 0. 이 트랙이 푸는 문제

준법팀의 매주 월요일 루틴:
1. 금융위·금감원·주요 매체에서 지난 주 규제·공시 뉴스를 훑는다.
2. 우리 조직에 영향 있는 것만 골라 분류한다.
3. 정해진 표 양식으로 리포트를 만들어 팀에 공유한다.

문제: 담당자마다 훑는 범위가 다르고, 표 형식이 매주 미묘하게 바뀌며, 바쁘면 건너뛴다.
→ **범위·형식·실행을 파일로 고정**하면 이 셋이 모두 사라집니다.

```
track-b-ops/
├── README.md
└── schema/regwatch.schema.json   ← 리포트 항목의 '형식 계약'
```
(상위 `codex-engineering/AGENTS.md`가 이 트랙에도 규율로 적용됩니다.)

---

## 1. 1단계 — `--search`로 라이브 조사

`--search`를 붙이면 Codex가 실시간 웹 검색을 씁니다. 읽기만 하므로 `read-only`가 안전합니다.

```bash
codex exec --search --sandbox read-only \
  --output-last-message /tmp/regwatch-draft.md \
  "당신은 은행 준법·리스크팀 애널리스트입니다. 최근 1주간 한국 금융 규제·공시 동향을
   조사하세요. 자본규제·소비자보호·AML/제재·공시 4개 영역을 훑고, 각 항목에 출처
   기관명과 발표일을 붙이세요. 확인 안 된 내용은 지어내지 말고 '[확인 필요]'로 표시."
```

**관찰 포인트**: AGENTS.md의 "출처 필수·추측 금지" 규칙을 프롬프트에 다시 쓰지 않아도
Codex가 지킵니다 — 규칙이 파일에 있기 때문입니다. 프롬프트가 짧아지는 만큼 실수가 줍니다.

---

## 2. 2단계 — `--output-schema`로 형식을 계약한다

조사 결과를 **매주 똑같은 구조**로 뽑는 것이 자동화의 핵심입니다. 자유 서술로 두면
담당자·주차마다 형식이 흔들립니다. JSON Schema로 형태를 강제합니다.

```bash
codex exec --search --sandbox read-only \
  --output-schema schema/regwatch.schema.json \
  --output-last-message /tmp/regwatch.json \
  "최근 1주 한국 금융 규제·공시를 조사해 schema에 맞춰 JSON으로 반환하세요.
   category와 impact는 스키마에 정의된 값 중에서만 고르세요."
```

`schema/regwatch.schema.json`은 이렇게 강제합니다:
- `category`는 `자본규제/소비자보호/AML제재/공시/기타` **다섯 값만** 허용 (오타·변형 불가)
- `impact`는 `높음/중간/낮음`
- `action`은 필수 — "검토 필요" 같은 빈 동사 대신 실행 가능한 다음 행동
- `additionalProperties: false` — 스키마에 없는 필드를 지어내지 못함

모델 출력이 이 형태를 벗어나면 **자동으로 재시도**됩니다. 결과가 항상 같은 스키마이므로
그대로 스프레드시트·대시보드·이메일에 흘려 넣을 수 있습니다.

**관찰 포인트**: 이것이 트랙 A의 "테스트=계약"에 대응하는 비개발 버전입니다. 코드가 아니라
**출력 형식**을 계약으로 고정했습니다. `codex/` 입문편의 "계약이 병렬성을 만든다"가 여기서는
"계약이 반복 가능성을 만든다"로 나타납니다.

## 3. 3단계 — 정례 자동 실행

한 번 완성한 명령을 스케줄러에 걸면 "매주 월요일 오전 8시 자동 발행"이 됩니다.

**Windows 작업 스케줄러** 또는 **cron**에 등록하거나, GitHub Actions로:

```yaml
# .github/workflows/regwatch.yml (개념 예시)
on:
  schedule:
    - cron: "0 23 * * 0"     # 매주 일요일 23:00 UTC = 월요일 08:00 KST
jobs:
  regwatch:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install -g @openai/codex
      - run: codex login --with-api-key <<< "${{ secrets.OPENAI_API_KEY }}"
      - run: |
          codex exec --search --sandbox read-only \
            --output-schema schema/regwatch.schema.json \
            --output-last-message regwatch.json \
            "최근 1주 한국 금융 규제·공시를 조사해 schema에 맞춰 JSON으로 반환"
      # 이후 regwatch.json을 메일/슬랙/시트로 배포하는 스텝 추가
```

여기까지 오면 만든 것은 "이번 주 리포트"가 아니라 **리포트를 만드는 시스템**입니다.

## 4. 교육 포인트

**① 코드를 안 짜도 하네스 엔지니어링은 가능하다.** 규칙(AGENTS.md)·형식 계약(스키마)·
자동 실행(스케줄)만으로 반복 업무가 시스템이 됩니다. "엔지니어링"은 프로그래밍이 아니라
**반복을 파일로 굳히는 사고방식**입니다.

**② 형식을 자유에 맡기지 마라.** 사람이 매번 표를 그리면 형식이 흔들립니다. `--output-schema`가
그 흔들림을 원천 차단합니다 — 비개발자가 얻을 수 있는 가장 강력한 통제 장치입니다.

**③ 읽기 전용이 조사 자동화의 기본.** 조사·리포트는 아무것도 고칠 필요가 없으니 항상
`--sandbox read-only`. 권한을 안 주는 것이 가장 확실한 안전입니다.

**④ 규제 도메인일수록 출처·추측 금지가 생명.** AGENTS.md에 이 규칙을 박아 두면 매 프롬프트가
짧아지고, 담당자가 바뀌어도 규율이 유지됩니다.

## 5. 심화 과제

1. **분류 확장**: 스키마의 `category` enum에 "지배구조"를 추가하고 재실행 — 프롬프트는
   그대로인데 산출 구조만 바뀌는 것(형식과 내용의 분리)을 확인하세요.
2. **Tool 결합**: `tool/`의 방식으로 금융 지표 계산 MCP(예: 자기자본비율)를 만들어, 규제
   항목에 "우리 조직 현재 수치"를 자동으로 붙이세요. 조사(모델) + 계산(Tool)의 결합입니다.
3. **실제 오픈소스 MCP 활용**: 금융감독원 FISIS를 감싼 오픈소스 MCP를 등록하면, 은행별
   재무지표를 직접 조회해 리포트에 근거로 넣을 수 있습니다. "만들기 전에 이미 있는 Tool을
   찾는다"의 실전입니다.
4. **크로스 플랫폼**: 같은 스키마·규칙을 Claude Code의 `.claude/`로 옮겨 `/harness`로 같은
   리포트를 만들어 보고, 무엇이 그대로 이식되고 무엇이 번역되는지 기록하세요.
