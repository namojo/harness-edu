# 트랙 B — 업무 자동화: 코드 없이 반복 업무를 시스템으로

> 코드를 짜지 않는 실무자를 위한 트랙입니다. 금융회사 **준법·리스크팀**이 매주 반복하는
> "규제·공시 모니터링 리포트"를 소재로, Codex를 **정례 업무 엔진**으로 쓰는 법을 배웁니다.
>
> 이 실습도 **Codex를 대화형으로** 진행합니다 — `codex`를 실행하거나 Codex 앱을 엽니다.
> 먼저 대화형으로 리포트를 만들어 보고(§1~§2), 그 흐름을 매주 자동으로 돌리는 법(§3)을
> 이어서 배웁니다.

트랙 A가 "코드를 고친다"였다면, 트랙 B는 "코드를 안 짜도 반복 업무를 파일로 굳힌다"입니다.
같은 원칙(상위 README) 위에서 도메인만 바뀝니다.

## 0. 이 트랙이 푸는 문제

준법팀의 매주 월요일 루틴:
1. 금융위·금감원·주요 매체에서 지난 주 규제·공시 뉴스를 훑는다.
2. 우리 조직에 영향 있는 것만 골라 분류한다.
3. 정해진 표 양식으로 리포트를 만들어 팀에 공유한다.

문제: 담당자마다 훑는 범위가 다르고, 표 형식이 매주 미묘하게 바뀌며, 바쁘면 건너뛴다.
→ **범위·형식·실행을 파일로 고정**하면 이 셋이 모두 사라집니다.

```
track-b-ops/
└── schema/regwatch.schema.json   ← (§3) 리포트 항목의 '형식 계약'
```
(상위 `codex-engineering/AGENTS.md`가 이 트랙에도 규율로 적용됩니다.)

이 폴더에서 Codex를 대화형으로 엽니다.

```bash
cd codex-engineering/track-b-ops
codex          # 또는 Codex 앱에서 이 폴더를 연다
```

## 1. 대화형으로 조사시키기

Codex 세션에 아래처럼 **입력**합니다. (웹 검색이 필요하면 세션에서 검색을 사용하도록
허용하세요 — CLI는 `codex --search`로 시작하면 라이브 검색이 켜집니다.)

```
너는 은행 준법·리스크팀 애널리스트야. 최근 1주간 한국 금융 규제·공시 동향을 조사해줘.
자본규제·소비자보호·AML/제재·공시 4개 영역을 훑고, 각 항목에 출처 기관명과 발표일을 붙여줘.
확인 안 된 내용은 지어내지 말고 '[확인 필요]'로 표시해줘.
```

**관찰 포인트**: AGENTS.md의 "출처 필수·추측 금지" 규칙을 프롬프트에 다시 쓰지 않아도
Codex가 지킵니다 — 규칙이 파일에 있기 때문입니다. 조사는 아무것도 고칠 필요가 없으니
Codex가 파일을 건드리지 않습니다(자연스럽게 읽기 전용).

## 2. 형식이 매주 흔들리는 문제를 만나기

이번엔 표로 정리해 달라고 해봅니다.

```
위 내용을 표로 정리해줘. 컬럼은 제목 / 출처 / 발표일 / 분류 / 영향도 / 다음 행동.
```

여러 번 실행하거나 담당자가 바뀌면, 분류 이름이 "자본규제"였다 "자본 규제"였다 하고,
영향도가 "높음"이었다 "상"이었다 흔들립니다. **자유 서술은 형식을 보장하지 못합니다.**
이 흔들림을 없애는 것이 §3입니다.

## 3. 형식을 계약으로 고정하고 자동화하기

매주 **똑같은 구조**를 보장하려면, 형식을 사람 지시가 아니라 **스키마 파일**로 강제합니다.
이는 대화형이 아니라 `codex exec` 자동화에서 `--output-schema`로 씁니다.

```bash
codex exec --search --sandbox read-only \
  --output-schema schema/regwatch.schema.json \
  --output-last-message regwatch.json \
  "최근 1주 한국 금융 규제·공시를 조사해 schema에 맞춰 JSON으로 반환하라.
   category와 impact는 스키마에 정의된 값 중에서만 고르라."
```

`schema/regwatch.schema.json`은 이렇게 강제합니다:
- `category`는 `자본규제/소비자보호/AML제재/공시/기타` **다섯 값만** 허용 (오타·변형 불가)
- `impact`는 `높음/중간/낮음`
- `action`은 필수 — "검토 필요" 같은 빈 동사 대신 실행 가능한 다음 행동
- `additionalProperties: false` — 스키마에 없는 필드를 지어내지 못함

모델 출력이 이 형태를 벗어나면 **자동으로 재시도**됩니다. 결과가 항상 같은 스키마이므로
그대로 스프레드시트·대시보드·이메일에 흘려 넣을 수 있습니다.

> `--output-schema`·`--output-last-message`는 **`codex exec` 전용**입니다(대화형 TUI에는
> 없습니다). "사람이 지켜보며 한 번" 할 땐 §1~§2 대화형으로, "형식을 보장하며 반복"할 땐
> 여기 자동화로 — 이 구분이 트랙 B의 핵심입니다.

### 매주 자동 실행

이 명령을 스케줄러(Windows 작업 스케줄러·cron)나 GitHub Actions에 걸면 "매주 월요일 오전
8시 자동 발행"이 됩니다.

```yaml
# .github/workflows/regwatch.yml (개념 예시)
on:
  schedule:
    - cron: "0 23 * * 0"     # 매주 일요일 23:00 UTC = 월요일 08:00 KST
jobs:
  regwatch:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install -g @openai/codex
      - run: codex login --with-api-key <<< "${{ secrets.OPENAI_API_KEY }}"
      - run: |
          codex exec --search --sandbox read-only \
            --output-schema schema/regwatch.schema.json \
            --output-last-message regwatch.json \
            "최근 1주 한국 금융 규제·공시를 조사해 schema에 맞춰 JSON으로 반환"
      # 이후 regwatch.json을 메일/슬랙/시트로 배포하는 스텝 추가
```

여기까지 오면 만든 것은 "이번 주 리포트"가 아니라 **리포트를 만드는 시스템**입니다.

## 4. 교육 포인트

**① 대화형으로 감을 잡고, 자동화로 형식을 보장한다.** §1~§2에서 형식이 흔들리는 문제를
직접 겪고, §3에서 스키마로 그 흔들림을 원천 차단합니다. 순서가 곧 학습입니다.

**② 형식을 자유에 맡기지 마라.** 사람이 매번 표를 그리면 형식이 흔들립니다. `--output-schema`가
비개발자가 얻을 수 있는 가장 강력한 통제 장치입니다.

**③ 읽기 전용이 조사의 기본.** 조사·리포트는 아무것도 고칠 필요가 없습니다. 대화형에서는
자연히 파일을 안 건드리고, 자동화에서는 `--sandbox read-only`로 명시합니다.

**④ 규제 도메인일수록 출처·추측 금지가 생명.** AGENTS.md에 이 규칙을 박아 두면 매 프롬프트가
짧아지고, 담당자가 바뀌어도 규율이 유지됩니다.

## 5. 심화 과제

1. **분류 확장**: 스키마의 `category` enum에 "지배구조"를 추가하고 §3을 재실행 — 프롬프트는
   그대로인데 산출 구조만 바뀌는 것(형식과 내용의 분리)을 확인하세요.
2. **Tool 결합**: `tool/`의 방식으로 금융 지표 계산 MCP(예: 자기자본비율)를 만들어, 규제
   항목에 "우리 조직 현재 수치"를 자동으로 붙이세요. 조사(모델) + 계산(Tool)의 결합입니다.
3. **실제 오픈소스 MCP 활용**: 금융감독원 FISIS를 감싼 오픈소스 MCP를 등록하면, 은행별
   재무지표를 직접 조회해 리포트에 근거로 넣을 수 있습니다.
4. **크로스 플랫폼**: 같은 스키마·규칙을 Claude Code의 `.claude/`로 옮겨 `/harness`로 같은
   리포트를 만들어 보고, 무엇이 그대로 이식되고 무엇이 번역되는지 기록하세요.
