# 트랙 A — 엔지니어링: Codex로 코드 유지보수·리뷰

> 제조 현장의 **품질 점검 스크립트**(`src/quality_check.py`)를 소재로, Codex를 실무
> 개발 흐름에 넣는 법을 배웁니다: 진단 → 패치 → 교차 리뷰. 이 스크립트에는 **의도된
> 결함 2개**가 숨어 있습니다. 초록불(테스트 통과) 뒤에 숨은 버그를 Codex가 잡아내는지
> 실험합니다.
>
> 이 실습은 **Codex를 대화형으로** 진행합니다 — 터미널에서 `codex`를 실행하거나 Codex 앱을
> 엽니다. 매번 셀을 붙여넣는 게 아니라, 세션 안에서 대화하듯 지시하고 승인합니다.
> (반복 작업을 자동화하는 `codex exec`·CI는 §5에서 따로 다룹니다.)

## 0. 예제 구성과 현재 상태 확인

```
track-a-engineering/
├── src/quality_check.py      ← 라인별 불량률(ppm) 점검 — 결함 2개 내장
├── tests/test_quality_check.py ← 통과하지만 결함을 못 잡는 테스트 (일부러)
├── data/production.csv        ← 라인 A·B·C 생산 데이터
└── .github/workflows/codex-review.yml  ← (§5) PR마다 자동 리뷰하는 CI
```

먼저 사람이 직접 현재 상태를 확인합니다.

```bash
cd codex-engineering/track-a-engineering
pip install pytest
python -m pytest tests/ -q          # → 3 passed  (초록불!)
python src/quality_check.py         # → 전 라인 '정상'으로 표시됨
```

테스트도 통과하고 출력도 멀쩡해 보입니다. **하지만 세 라인 모두 실제 불량률은 ppm 기준
위험 수준**입니다(C라인 16,393 ppm). 여기서 트랙 A가 시작됩니다.

이제 이 폴더에서 Codex를 대화형으로 엽니다.

```bash
codex          # 터미널 TUI. 또는 Codex 앱에서 이 폴더를 연다.
```

## 1. 진단 — "고치지 말고 검토만"

Codex 세션에 아래처럼 **입력**합니다. 수정하지 말라고 명시하면 Codex는 파일을 읽기만 합니다.

```
src/quality_check.py 를 검토해줘. 테스트는 통과하지만 불량률(ppm) 계산과 경계 처리에
결함이 있는지, 있다면 무엇인지 근거와 함께 진단해줘. 지금은 수정하지 말고 진단만.
```

**관찰 포인트**: 테스트가 통과했는데도(§0의 `3 passed`) 진단이 결함을 지적한다면,
"테스트 통과 ≠ 정확"이 증명된 것입니다. 이것이 검증을 테스트와 **별도로** 두는 이유입니다.

> 승인·권한은 대화형에서 자동으로 통제됩니다. Codex는 파일을 **고치거나 명령을 실행하기
> 전에 승인을 묻습니다.** "진단만" 단계에서는 애초에 쓰기를 요청하지 않으므로 안전합니다.

---

## 2. 패치 — 승인하며 고치기

진단이 타당하면, 이번엔 수정을 지시합니다.

```
방금 진단한 결함 두 가지(불량률 단위 오류, 0 나눗셈 경계)를 수정해줘. 그리고 tests/ 에
절대값·경계값 테스트를 추가해서 이 결함을 실제로 잡도록 강화하고, pytest로 통과를 확인해줘.
무엇을 왜 바꿨는지 요약해줘.
```

Codex가 파일을 고치거나 `pytest`를 실행하기 전에 **승인을 요청**합니다. 변경 내용을 보고
승인/거부하세요 — 이 승인 지점이 "사람이 개입하는 안전장치"입니다. 다 끝나면 diff를 눈으로
한 번 더 확인합니다.

---

## 3. 교차 검증 — 분리된 눈으로

구현한 그 세션이 자기 결과를 최종 승인하지 않게 합니다. 두 가지 방법이 있습니다.

- **같은 세션에서 관점을 바꿔 부탁**: 
  ```
  방금 수정을 리뷰어 관점에서 다시 검토해줘. 놓친 경계값이나 회귀 위험이 없는지,
  추가한 테스트가 정말 원래 버그를 잡는지 비판적으로 봐줘.
  ```
- **전용 리뷰 명령 `codex review`** (변경분을 정형화된 포맷으로 리뷰):
  ```bash
  codex review --uncommitted        # 방금 만든 수정(diff)을 리뷰
  ```
  `codex review`는 **변경분(diff)** 을 대상으로 하므로, §2에서 수정을 만든 **뒤**에
  실행해야 리뷰할 것이 생깁니다. 갓 클론해 변경이 없는 상태에서는 리뷰 대상이 없습니다.

**관찰 포인트**: 잘 쓰는 것과 빠뜨리지 않는 것은 다른 능력입니다. 구현자 ≠ 최종 승인자로
분리하는 것이 이 단계의 핵심입니다.

## 4. 정답 — 숨어 있던 결함 2개

스스로/Codex로 찾아본 뒤 펼쳐 보세요.

<details>
<summary>결함 공개</summary>

1. **단위 오류** (`defect_ppm`): `defects / total_count * 1000` — ppm은 백만 개당 수이므로
   `* 1_000_000`이어야 합니다. `* 1000`은 ‰(퍼밀) 수준이라, 실제 9,950~16,393 ppm인
   불량률이 10~16으로 표시됩니다. 그 결과 `PPM_THRESHOLD = 12000` 경보가 **영원히
   울리지 않습니다.** 위험한 라인이 "정상"으로 통과하는, 조용하지만 치명적인 버그입니다.

2. **0 나눗셈 경계**: `total_count == 0`(생산 없는 날/빈 행)이면 `defects / total_count`가
   `ZeroDivisionError`로 죽습니다. 가드가 없습니다.

**왜 테스트가 못 잡았나**: 테스트는 (a) 키·개수 등 구조, (b) 라인 간 상대 크기만 검사합니다.
단위가 틀려도 순위는 같고, 빈 행 케이스는 아예 없습니다. **초록불은 "검사한 것이 맞다"는
뜻이지 "코드가 맞다"는 뜻이 아닙니다.**

</details>

## 5. (심화) 반복되면 자동화한다 — `codex exec`와 CI

위 1~3단계는 사람이 지켜보는 **대화형**이었습니다. 이 흐름을 매일·매 PR마다 반복한다면,
그때 비로소 **비대화형 자동화**로 굳힙니다. 여기서부터가 하네스 엔지니어링의 자동화 영역입니다.

### 5.1 `codex exec` — 사람 없이 한 번에

`codex exec`는 대화형 TUI 없이 프롬프트 하나를 실행하는 자동화용 명령입니다.

```bash
# 진단(읽기 전용). 결과를 파일로 받는다.
codex exec --sandbox read-only --output-last-message /tmp/diag.md \
  "src/quality_check.py의 불량률 계산·경계 처리 결함을 진단하라. 수정하지 마라."
```

- **핵심 차이**: 대화형과 달리 `codex exec`는 **승인을 묻지 않습니다.** 그래서 위험 통제를
  오직 `--sandbox`로 합니다 — 분석은 `read-only`, 패치는 `workspace-write`.
- 인터랙티브 전용 플래그 `--ask-for-approval`을 `codex exec`에 붙이면 **에러**입니다.
  (승인 개념 자체가 없기 때문)
- 최종 답은 `--output-last-message`로 파일에 분리 수집하세요. stdout에는 진행 로그가 섞입니다.

### 5.2 CI 게이트

`.github/workflows/codex-review.yml`은 PR이 열릴 때마다 ① 테스트를 돌리고 ② `codex review`로
자동 리뷰해 결과를 아티팩트로 남깁니다. 사람이 리뷰를 깜빡해도 파이프라인이 **검증자 분리
원칙을 강제**합니다.

실사용 준비:
- 저장소 **Settings → Secrets and variables → Actions**에 `OPENAI_API_KEY` 등록
- 조직 정책상 CI에서 외부 모델 호출이 가능한지 먼저 확인

## 6. 교육 포인트

**① 초록불을 믿지 마라.** 통과하는 테스트는 "검사 대상만" 보증합니다. 단위·경계·누락은
테스트가 없으면 통과가 곧 위험 신호입니다. 그래서 리뷰가 테스트와 별개로 필요합니다.

**② 대화형에선 승인이 안전장치.** Codex는 파일을 고치기 전에 묻습니다(§1~§2). "진단만"으로
시작해 변경을 승인하며 진행하는 것이, 처음부터 전권을 주는 것보다 안전하고 학습에도 좋습니다.

**③ 구현자 ≠ 승인자.** 구현한 세션이 자기 결과를 최종 승인하지 않고, 관점을 바꾼 리뷰나
`codex review`로 교차 검증합니다(§3). `harness/`의 compliance-reviewer 분리와 같은 원리입니다.

**④ 대화형으로 익히고, 반복되면 자동화.** 손에 익힌 흐름만 `codex exec`·CI로 굳힙니다(§5).
자동화는 대화형의 대체가 아니라 "충분히 반복되는 일"에 대한 졸업입니다.

## 7. 심화 과제

1. **회귀 방지**: 패치 후 추가된 테스트가 원래 버그(×1000, 0 나눗셈)를 정말 잡는지,
   버그를 일부러 되돌려 확인하세요(빨간불이 떠야 정상).
2. **두 번째 의견**: Claude Code로 같은 파일을 진단시켜 Codex의 진단과 비교해 보세요.
   서로 다른 모델의 교차 검증이 자기 확증을 줄입니다.
3. **Tool 연결**: `tool/`의 `production-metrics` MCP를 이 스크립트 대신 써서, 계산을 Tool에
   위임하고 스크립트는 임계치 판정·리포트만 하게 리팩터하세요. "계산은 Tool, 판정은 코드"의
   경계가 잡힙니다.
4. **자동화 졸업**: §1~§3 흐름을 `codex exec`(§5.1)로 재현하고, CI(§5.2)에 걸어 매 PR마다
   자동 리뷰되게 만들어 보세요.
