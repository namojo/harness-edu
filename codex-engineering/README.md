# Codex 기반 Harness Engineering — 실무편

> `codex/` 폴더가 "코딩 몰라도 따라오는" 트렌드 보고서 입문이었다면, 여기는 **실무에서
> 매일 돌리는** Codex 하네스를 다룹니다. Codex를 "가끔 질문하는 챗봇"이 아니라
> **규칙·샌드박스·자동화가 갖춰진 엔지니어링 도구**로 쓰는 법입니다.
>
> 두 트랙으로 나뉩니다 — 코드를 다루는 사람을 위한 **A. 엔지니어링 트랙**,
> 코드를 안 짜는 실무자를 위한 **B. 업무 자동화 트랙**. 필요한 쪽부터 보세요.

---

## 0. Harness Engineering이란 무엇인가

하네스(harness)는 에이전트가 **안전하고 반복 가능하게** 일하도록 감싸는 뼈대입니다.
"좋은 프롬프트를 쓴다"와 "하네스를 설계한다"의 차이는 이겁니다:

| | 프롬프트 잘 쓰기 | 하네스 엔지니어링 |
|---|---|---|
| 단위 | 한 번의 대화 | 반복되는 작업 흐름 |
| 재현성 | 매번 다름 | 누가 돌려도 같음 |
| 안전장치 | 사람이 눈으로 검토 | 샌드박스·승인정책·검증자로 강제 |
| 자산 | 머릿속/채팅 로그 | 파일(AGENTS.md·에이전트·스키마·CI) |

### Codex Harness Engineering 5원칙

1. **대화형으로 익히고, 반복되면 자동화한다** — 처음엔 `codex`(또는 Codex 앱) 대화형으로
   일을 시키며 승인·권한 감각을 익힌다. 세 번 이상 반복되는 일만 `codex exec` 비대화형으로
   굳힌다. 자동화는 대화형의 대체가 아니라 "충분히 반복되는 일"에 대한 졸업이다.
2. **권한은 최소로** — 대화형에선 Codex가 쓰기 전에 승인을 묻는다. 자동화(`codex exec`)에선
   승인 개념이 없으므로 `--sandbox`로 통제한다: 분석·리뷰는 `read-only`, 패치만
   `workspace-write`. `danger-full-access`/`--yolo`는 격리 환경 밖에서 금지.
3. **규칙은 파일로** — 반복되는 원칙은 프롬프트가 아니라 `AGENTS.md`에 적는다.
   Codex가 시작할 때 자동으로 읽는다.
4. **출력은 계약으로** — 파이프라인에 넘길 결과는 `--output-schema`로 형태를 강제하고
   `--output-last-message`로 최종 답만 분리 수집한다. stdout 파싱 금지.
5. **검증자는 분리** — 만든 사람이 자기 결과를 검사하지 않는다. `codex review`나
   별도 리뷰어 에이전트로 교차 검증한다.

## 1. 준비

```bash
codex --version          # 설치 확인 (이 교안은 codex 0.128+ 기준)
codex login status       # 인증 상태 — 'logged in'이면 OK
# 실패 시:  codex login   (ChatGPT OAuth 권장)  또는  codex login --device-auth
```

```
codex-engineering/
├── README.md                    ← 본 교안 (공통 개념 + 트랙 지도)
├── AGENTS.md                    ← 엔지니어링 공통 규율 (Codex가 자동으로 읽음)
├── .codex/
│   ├── config.example.toml      ← 프로파일·샌드박스·MCP 설정 예시
│   └── agents/                  ← 재사용 에이전트 정의
│       ├── implementer.toml     (구현 담당)
│       └── reviewer.toml        (검증 담당 — 분리된 눈)
├── track-a-engineering/         ← A. 코드 실무 (제조 품질 스크립트)
└── track-b-ops/                 ← B. 비개발 업무 자동화 (금융 규제 모니터링)
```

`config.example.toml`은 **자동화 단계에서 선택적으로** 씁니다 — `~/.codex/config.toml`에
반영하면 `-p` 프로파일(자동화용 옵션 묶음)과 MCP 서버 등록이 재사용됩니다. 대화형 실습에는
필요 없습니다. (기존 설정이 있으면 필요한 블록만 병합하세요.)

## 2. 트랙 지도 — 어느 쪽부터?

| | **트랙 A · 엔지니어링** | **트랙 B · 업무 자동화** |
|---|---|---|
| 대상 | 코드를 다루는 개발·데이터 담당 | 코드를 안 짜는 기획·관리·리서치 담당 |
| 시나리오 | 제조: 생산 품질 스크립트 유지보수 | 금융: 규제·공시 모니터링 리포트 |
| 배우는 것 | 대화형 진단·패치·교차리뷰 → (자동화) `codex exec`·CI | 대화형 조사 → (자동화) `--output-schema`·스케줄 |
| Codex의 강점 | 자기 코드 리팩터·테스트 통과까지 | 반복 리서치를 파일 계약으로 시스템화 |
| 폴더 | [`track-a-engineering/`](track-a-engineering/) | [`track-b-ops/`](track-b-ops/) |

두 트랙 모두 **같은 5원칙** 위에서 돌아갑니다. 도메인과 산출물만 다릅니다.

## 3. 권한 통제 — 대화형과 자동화가 다르다 (양 트랙 공통, 반드시 숙지)

Codex 실무의 안전은 여기서 결정됩니다. **모드에 따라 통제 방식이 다릅니다.**

**대화형 (`codex` TUI / Codex 앱) — 학습·일회성의 기본.**
Codex는 파일을 고치거나 명령을 실행하기 **전에 승인을 묻습니다.** "진단만 해줘"라고 하면
읽기만 하고, 수정을 지시하면 변경안을 보여 주고 승인을 요청합니다. 사람이 그 자리에서
허용/거부하므로, 승인 자체가 안전장치입니다. (승인 강도는 `--ask-for-approval`로 조절)

**자동화 (`codex exec`) — 반복을 굳힐 때.**
사람이 없으므로 **승인 개념이 없습니다.** 위험 통제를 오직 `--sandbox`로 합니다.

| Sandbox | 쓰기 권한 | 적합한 작업 |
|---|---|---|
| `read-only` | 없음 | 분석·리뷰·조사·리포트 (자동화 기본값) |
| `workspace-write` | 작업 루트 + `--add-dir` | 패치 생성·파일 산출 |
| `danger-full-access` | 전체 | ❌ 격리 환경 외 금지 |

> **핵심 함정**: 인터랙티브 전용 플래그 `--ask-for-approval`을 `codex exec`에 붙이면 **즉시
> 에러**로 종료합니다. 승인 개념이 없는 모드이기 때문입니다. 대화형은 승인, 자동화는 샌드박스 —
> 섞지 마세요.

## 4. 교육 포인트 (공통)

**① "AI에게 시킨다"에서 "AI 작업 흐름을 설계한다"로.** 실무의 차이는 모델 성능이 아니라
그 주변의 뼈대 — 규칙 파일, 샌드박스, 검증 단계, 자동 실행 — 에서 납니다.

**② 안전은 신뢰가 아니라 구조로.** "알아서 잘하겠지"가 아니라 `read-only`로 권한을 뺏어
안전을 보장합니다. 권한을 준 만큼만 위험합니다.

**③ 반복되면 파일로 굳힌다.** 세 번 이상 하는 일은 AGENTS.md·프로파일·스키마·CI로
자산화합니다. 이 과정 전체를 관통하는 "프롬프트 밖으로 꺼내기"의 Codex 실무 버전입니다.

**④ Claude Code와 같은 원리, 다른 문법.** `harness/`가 `.claude/`로 했던 것을 Codex는
`AGENTS.md` + `.codex/`로 합니다. 개념(규칙·역할·검증 분리)은 이식되고 문법만 번역됩니다.
