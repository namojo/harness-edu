먼저 알고 시작할 것

모든 실습을 관통하는 4가지 원리

도구가 Claude Code든 Codex든, 업종이 제조든 금융이든 — 좋은 하네스는 늘 이 네 가지의 변주입니다.

01

역할 분리

"무엇을(스펙)·누가(에이전트)·어떻게(스킬)"를 한 프롬프트에 뭉치지 않고 파일로 나눕니다. 나눠야 재사용·비교·유지보수가 됩니다.

02

지식의 외부화

매핑 규칙·도메인 지식·표준 양식을 프롬프트가 아니라 파일로 꺼냅니다. 지식이 바뀌어도 로직은 그대로입니다.

03

계약이 병렬성을 만든다

표준 스키마·출력 형식을 먼저 정하면 일을 병렬로 나누고, 매번 같은 형식으로 반복할 수 있습니다.

04

검증자는 분리한다

만든 주체가 자기 결과를 최종 승인하지 않습니다. 별도 리뷰어가 교차 검증합니다. 잘 쓰기와 빠뜨리지 않기는 다른 능력입니다.

하네스의 구성 요소

Agent · Skill · Tool — 세 개의 축

하네스는 결국 이 세 가지의 조합입니다. 무엇을 언제 쓰는지 구분하는 것이 이 과정의 핵심 감각입니다.

Agent
누가 — 역할·경계
  • 일을 나눠 맡길 때
  • 지시문(정의 파일)로 존재
  • 예: 요구분석가, 검증자
Skill
무엇을·어떻게 — 지식·절차
  • 판단·문체·도메인 지식
  • 프롬프트에 주입되는 문서
  • "설명서" — 모델이 해석
Tool
무엇으로 실행 — 결정론적
  • 계산·조회·외부 연동
  • 실제 코드(예: MCP 서버)
  • "계산기" — 항상 같은 값

판단 규칙 — 정의가 고정돼 늘 같은 답이 나와야 하는 일(계산·조회·API)은 Tool로, 맥락에 따라 달라지는 판단(무엇이 중요한가·어떻게 쓸까)은 Skill로. 그 둘을 누구에게 맡길지가 Agent입니다.

두 개의 에이전트, 하나의 원리

대화형으로 익히고,
반복되면 자동화한다

Claude Code와 Codex는 문법이 다를 뿐 개념은 같습니다. 처음엔 대화형으로 일을 시키며 승인·권한 감각을 익히고, 세 번 이상 반복되는 일만 비대화형으로 자동화합니다. 자동화는 대화형의 대체가 아니라, 충분히 반복되는 일에 대한 졸업입니다.

대화형codex(TUI)·Codex 앱·Claude Code. 쓰기 전에 승인을 묻습니다 — 승인이 안전장치. 자동화codex exec·CI. 승인이 없으므로 샌드박스(read-only/workspace-write)로 통제. 공통규칙은 AGENTS.md·CLAUDE.md로, 출력은 스키마로 계약.
트랙 A — 대화형 → 자동화
# 1) 대화형: 권한 없이 진단만 ▸ quality_check.py 를 검토해줘. 수정 말고 결함만 진단. ✓ 단위 오류(ppm ×1000)·0 나눗셈 발견 # 2) 승인하며 패치 (쓰기 전 승인 요청) approve? edit quality_check.py [y] # 3) 반복되면 자동화 (CI 게이트) $ codex review --uncommitted
