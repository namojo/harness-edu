# 정책 데이터 → 정부 표준 슬라이드 — 하네스(Harness) 실습

> 공개 정책 데이터(대한민국 AI 행동계획 2026)를 수집·정규화하여 엑셀 현황판을 갱신하고,
> **대한민국 정부 디자인 표준(KRDS) 테마의 보고용 슬라이드**를 자동 생성하는
> [revfactory/harness](https://github.com/revfactory/harness) 스타일 하네스 구성 교안입니다.

## 1. 문제 정의

정책 이행현황 보고는 전형적인 반복 지식노동입니다. ① 흩어진 원천 데이터(공식 문서 + 분석 대시보드)를 수집하고, ② 기준일에 맞춰 현황 지표를 재계산하고, ③ 조직 표준 디자인의 슬라이드로 재작성합니다. 매주·매월 같은 구조가 반복되지만, 데이터 소스가 비정형(웹 대시보드, PDF, JS 임베디드 JSON)이라 단순 매크로로는 자동화가 어렵습니다.

이 실습은 그 전 과정을 **"에이전트(누가) + 스킬(어떻게)" 분리 원칙의 하네스**로 구성합니다. `xlsx/` 실습이 단일 프롬프트 ETL이었다면, 이번에는 수집 → 갱신 → 디자인 생성 → 검수로 이어지는 **파이프라인형 멀티 스텝 하네스**입니다.

## 2. 소재: 대한민국 AI 행동계획 2026

| 항목 | 내용 |
|---|---|
| 근거 | AI기본법 제6조 법정 기본계획, ’26. 2. 25. 국가AI전략위 제2차 전체회의 의결 |
| 구조 | 3대 정책축 · 12대 전략분야 · **99개 실천과제 · 326개 정책권고** |
| 데이터 소스 | [K-AI2026 대시보드 (UNOFFICIAL)](https://hollobit.github.io/K-AI2026/) — 과제·권고 구조화 데이터, D-Day 기반 위험도 분류 로직 |

> ⚠️ 대시보드는 개인이 만든 비공식 분석 자료이며, 본 실습 산출물 역시 교육용 UNOFFICIAL 자료입니다. 정부 공식 이행평가가 아닙니다.

## 3. 파일 구성

| 파일 | 역할 |
|---|---|
| `AI행동계획.xlsx` | **데이터 계층**. `★최종안` 시트(326개 권고 원문) + 이행현황 컬럼(기한일자·D-Day·위험도·이행상태, 전부 **수식**) + `이행현황요약` 시트(기준일 셀 기반 자동 집계) |
| `AI행동계획_이행현황.pptx` | **산출물**. KRDS 테마 9장 보고 슬라이드 (기준일 2026. 7. 3.) |
| `README.md` | 본 교안 |

핵심 설계 — **기준일 하나만 바꾸면 전체가 갱신되는 살아있는 문서**: `이행현황요약!B3`(기준일)을 수정하면 326건 각각의 D-Day와 위험도(기한초과/긴급/주의/관찰/여유), 모든 집계가 수식으로 재계산됩니다. 에이전트가 값을 하드코딩했다면 매번 재작업이 필요했을 것입니다.

## 4. 하네스 구성 (Agent × Skill)

revfactory/harness의 원칙에 따라 에이전트 정의(`.claude/agents/`)와 스킬(`.claude/skills/`)을 분리하고, **파이프라인 패턴**으로 연결합니다.

```
[data-collector] ──▶ [xlsx-updater] ──▶ [slide-designer] ──▶ [qa-reviewer]
     수집·정규화          현황 재계산           KRDS 슬라이드 생성        시각 검수
```

| 에이전트 | 역할 | 사용 스킬 |
|---|---|---|
| `data-collector` | 대시보드/저장소에서 과제·권고 데이터 추출, 스키마 검증 (99과제/326권고 카운트 일치 확인) | web-fetch, 데이터 파싱 |
| `xlsx-updater` | 기준일 기반 D-Day·위험도 수식 갱신, 요약 시트 재계산, 수식 오류 0건 검증 | `xlsx` 스킬 |
| `slide-designer` | 요약 데이터를 KRDS 테마 슬라이드로 변환 (차트는 네이티브·편집 가능) | `pptx` 스킬 + KRDS 디자인 규칙 |
| `qa-reviewer` | 슬라이드 렌더링 이미지 검수 (오버플로·겹침·수치 불일치), 데이터-슬라이드 교차 검증 | 이미지 분석 |

에이전트 팀 모드에서는 `qa-reviewer`가 발견한 결함을 `slide-designer`에게 직접 전달(SendMessage)하여 수정 루프를 자체 조율합니다.

### KRDS 디자인 규칙 (slide-designer 스킬에 내장할 지식)

- **팔레트**: Primary `#246BEB`(KRDS 파랑) 지배 + 심연 남색 `#0B2E5C`(표지·클로징) + 위험 `#DE3412` / 주의 앰버 / 여유 초록은 상태 표시에만 제한 사용
- **구조**: 다크 표지 → 라이트 본문 → 다크 클로징 (샌드위치), 좌상단 제목 + 한 줄 핵심 메시지 고정
- **데이터 우선**: 큰 숫자 스탯 카드 + 네이티브 차트(도넛·막대·누적바), 표는 남색 헤더
- **정직성 장치**: 모든 장에 기준일 푸터, ‘기한초과 ≠ 미이행’ 해석 주의 문구, 클로징에 출처·한계 명시

## 5. 실행 프롬프트

Claude Cowork 또는 Claude Code에서 이 디렉토리를 열고 다음을 실행합니다.

```
당신은 AI 정책 이행현황 보고 하네스의 오케스트레이터입니다.

[입력]
- slide/AI행동계획.xlsx (326개 정책권고 + 이행현황 수식)
- K-AI2026 대시보드(hollobit.github.io/K-AI2026)의 과제 구조화 데이터

[작업 절차]
1. 수집: 대시보드 데이터에서 과제 수(99)·권고 수(326)가 엑셀과 일치하는지
   검증한다. 불일치 시 [데이터 이상]으로 보고하고 중단한다.
2. 갱신: 이행현황요약!B3의 기준일을 오늘 날짜로 변경하고 전체 수식을
   재계산한다. 수식 오류(#REF!, #VALUE! 등)는 0건이어야 한다.
3. 집계 확인: 위험도별(기한초과/긴급/주의/관찰/여유/기한없음) 건수와
   비중, 정책축별·부처별·분기별 분포를 추출한다.
4. 슬라이드 생성: KRDS 테마(Primary #246BEB, 남색 #0B2E5C)로 9장 내외의
   보고 슬라이드를 생성한다. 차트는 이미지가 아닌 네이티브 차트로 만들고,
   모든 수치는 엑셀 집계값과 일치해야 한다.
5. 검수: 슬라이드를 이미지로 렌더링하여 텍스트 오버플로·요소 겹침·
   수치 불일치를 점검하고, 발견 시 수정 후 1회 재검수한다.

[불변 규칙]
- 수치는 엑셀이 유일한 진실 공급원(Single Source of Truth)이다.
  슬라이드에 임의 수치를 쓰지 않는다.
- '기한초과'는 기한 도래를 의미할 뿐 미이행 확정이 아니므로,
  해석 주의 문구를 반드시 포함한다.
- UNOFFICIAL 출처와 한계를 클로징 슬라이드에 명시한다.
```

## 6. 검수 체크리스트

- [ ] 엑셀 수식 오류 0건, 기준일 변경 시 위험도 자동 재계산 확인
- [ ] 슬라이드 수치 = 엑셀 집계값 (권고 326 / 위험도 합계 326 / 정책축 134+114+78)
- [ ] 차트가 이미지가 아닌 편집 가능한 네이티브 차트인가
- [ ] KRDS 팔레트 준수 (파랑 지배, 상태색 제한 사용, 다크-라이트-다크 구조)
- [ ] ‘기한초과 ≠ 미이행’ 해석 주의 문구와 UNOFFICIAL 출처 명시 여부
- [ ] 텍스트 오버플로·요소 겹침 없음 (렌더링 이미지 기준)

## 7. 교육 포인트

**① 데이터 계층과 표현 계층의 분리.** 엑셀이 SSOT(단일 진실 공급원), 슬라이드는 뷰(View)입니다. 에이전트에게 "슬라이드 수치는 반드시 엑셀에서"라는 불변 규칙을 주면, LLM이 그럴듯한 숫자를 지어내는 것을 구조적으로 차단합니다.

**② 수식은 에이전트가 남기는 재사용 가능한 로직.** D-Day·위험도를 값이 아닌 수식으로 남겼기 때문에, 다음 보고 주기에는 기준일 셀 하나만 바꾸면 됩니다. 에이전트의 산출물이 '일회성 답변'이 아니라 '갱신 가능한 시스템'이 되는 지점입니다.

**③ 도메인 지식의 스킬화.** KRDS 팔레트, 위험도 분류 기준(D<0/≤34/≤120/≤240), 해석 주의 문구 같은 도메인 규칙을 스킬 문서로 외부화하면, 어떤 에이전트가 실행해도 같은 품질이 재현됩니다 — `xlsx/` 실습의 표준용어집과 같은 원리입니다.

**④ 검증 없는 자동화는 자동화된 오보.** 정책 수치가 틀린 보고서는 없느니만 못합니다. 수집 단계의 카운트 일치 검증, 갱신 단계의 수식 오류 검증, 검수 단계의 수치 교차 검증 — 3중 게이트가 파이프라인의 신뢰를 만듭니다.

## 8. 심화 과제

1. **하네스 실제 구축**: `/harness` 스킬로 위 4개 에이전트 정의와 스킬 파일을 실제 생성하고, 에이전트 팀 모드로 전체 파이프라인을 실행해 보세요.
2. **주기 실행**: 공식 이행실적이 공개되면 `이행상태` 컬럼을 실적 데이터와 연동하고, 매주 기준일을 갱신하는 배치 잡으로 발전시켜 보세요.
3. **부처별 개별 보고**: 팬아웃 패턴으로 부처별 1장 요약 슬라이드 18종을 병렬 생성해 보세요.
4. **xlsx 실습과 연결**: `xlsx/`의 표준용어집 패턴을 이 도메인에 적용해 '정책 용어 표준화 스킬'을 추가해 보세요.
