# 하네스(Harness) 자가 학습 가이드

> 이 저장소의 모든 실습을 **혼자, 순서대로** 따라 할 수 있도록 만든 길잡이입니다.
> 각 폴더의 README가 "깊이"라면, 이 문서는 "경로"입니다 — 무엇을 어떤 순서로,
> 무엇을 관찰하며, 어디까지 됐으면 다음으로 넘어갈지를 안내합니다.
>
> 대상: 제조·금융 등 현업 실무자. **코딩 지식은 필요 없습니다** (확장 단계만 Python 조금).

## 0. 이 가이드 사용법

- 각 모듈은 **목표 → 준비 → 따라하기 → 관찰 포인트 → 자가 점검 → 다음**의 6단 구성입니다.
- **자가 점검(✅)** 항목을 다 채우면 그 모듈을 이해한 것입니다. 못 채우면 해당 폴더 README로
  돌아가세요.
- 명령은 **그대로 복사해 실행**할 수 있게 적었습니다. 경로·날짜는 각자 환경에 맞게 바꾸세요.
- 전체를 다 할 필요는 없습니다. **입문 3개**만 해도 하네스의 핵심은 잡힙니다. 실무·확장은
  필요할 때 골라 하세요.

## 3. 먼저 알고 시작할 4가지 관통 원리

모든 실습은 아래 원리의 변주입니다. 실습 중 이 원리들이 어떻게 나타나는지 찾아보세요.

1. **What · Who · How 분리** — "무엇을(스펙)", "누가(에이전트)", "어떻게(스킬)"를 한
   프롬프트에 뭉치지 않고 파일로 나눈다. 셋을 나눠야 재사용·비교·유지보수가 된다.
2. **지식의 외부화** — 매핑 규칙·도메인 지식·표준 양식을 프롬프트가 아니라 파일
   (용어집·스킬·스키마)로 꺼낸다. 지식이 바뀌어도 로직은 그대로.
3. **계약이 병렬성·반복성을 만든다** — 표준 스키마/출력 형식을 먼저 정하면 작업을
   병렬로 나누고, 매번 같은 형식으로 반복할 수 있다.
4. **검증자는 분리한다** — 만든 주체가 자기 결과를 최종 승인하지 않는다. 별도 리뷰어·
   `codex review`·QA 에이전트가 교차 검증한다. 잘 쓰는 것과 빠뜨리지 않는 것은 다른 능력.

## 4. 모듈별 자가 학습

### 📗 입문 1 — `xlsx/` 제조 공정 보고서 표준화

- **목표**: 용어·문체가 제각각인 6개 분임조 보고서를 표준용어집 기반으로 통합.
- **준비**: `cd xlsx` 후 Claude Code(또는 Codex) 실행.
- **따라하기**: `prompt.md`의 내용을 그대로 입력 → 에이전트가 6개 엑셀을 읽고 정규화해
  `통합_생산보고서.xlsx` 생성.
- **관찰 포인트**: 치환 규칙이 프롬프트가 아니라 `공정_표준용어집.xlsx`에 있다(원리 2).
  용어집에 없는 은어는 `[미확인 용어]`로 에스컬레이션한다("모르면 멈춰라").
- **자가 점검**
  - [ ] 은어(빵꾸·데나오시 등)가 표준용어로 치환됐다
  - [ ] 수치가 원본과 100% 일치한다 (표현만 바꾸고 숫자는 불변)
  - [ ] 합계·불량률이 하드코딩이 아니라 **수식**으로 계산됐다
- **다음**: 지식을 파일로 뺐다는 감각을 갖고 codex/로.

### 📗 입문 2 — `codex/` 세 가지 일하는 방식

- **목표**: 같은 보고서를 단일→멀티→하네스 3단계로 만들며 "이 일은 몇 단계짜리인가"를 익힌다.
- **준비**: `cd codex`, `codex login status`로 로그인 확인.
- **따라하기** (자세히는 [`codex/README.md`](codex/README.md)):
  ```bash
  # Level 1 — 단일 에이전트
  codex exec --search "전략마케팅팀 리서처로서 '디스플레이 × AI 트렌드' 보고서를
  report/trend-brief.md 로 작성하라. 모든 수치에 출처와 날짜를 붙여라."
  # Level 2 — 대화형(codex)에서 서브에이전트 3인 병렬 조사 → report-writer 통합 → fact-checker 검증
  # Level 3 — 자산을 활용한 한 줄 실행
  codex exec --search "trend-report 스킬과 .codex/agents/ 팀으로 이번 달 보고서를 report/2026-07.md 로 생산하라"
  ```
- **관찰 포인트**: L1은 속도, L2는 깊이·검증(집필자≠검증자, 원리 4), L3은 재현성에서 이긴다.
- **자가 점검**
  - [ ] L1과 L3의 실행 명령이 둘 다 한 줄이지만, L3이 기대는 **자산(파일)의 양**이 다름을 이해했다
  - [ ] 서브에이전트는 서로 대화하지 않고 부모가 산출물을 넘긴다는 것을 관찰했다(원리 3)
- **다음**: 같은 A/B 비교를 Claude Code로 하는 harness/로.

### 📗 입문 3 — `harness/` 단일 프롬프트 vs 에이전트 팀

- **목표**: 배경지식 없는 상태에서 전문가 에이전트 구조가 무엇을 바꾸는지 A/B로 확인.
- **준비**: `harness/` 폴더를 Claude Code로 열기 (`CLAUDE.md`·`.claude/`가 자동 인식됨).
- **따라하기**:
  ```
  # 트랙 A (단일) — 별도 세션에서
  첨부한 과업지시서를 기반으로 사업 제안서를 docx로 작성해줘.
  # 트랙 B (하네스)
  /harness 첨부한 과업지시서를 기반으로 사업 제안서 docx를 작성하고자 합니다.
  ```
- **관찰 포인트**: B의 우위는 문장이 아니라 중간 산출물 `artifacts/요구사항_인벤토리.md`
  하나에서 파생된다(원리 1·2). compliance-reviewer가 조건 조항 누락을 잡는다(원리 4).
- **자가 점검**
  - [ ] `제안서A`와 `제안서B`를 나란히 열고 4개 비교 지점의 차이를 확인했다
  - [ ] "A는 요약한 제안서, B는 응답한 제안서"의 의미를 설명할 수 있다
- **다음**: 입문 선택 실습(slide/·map/·dashboard/)을 하거나, 바로 실무 트랙으로.

### 📘 입문 선택 — `slide/` · `map/` · `dashboard/`

세 실습 모두 파이프라인/팬아웃형 하네스입니다. 각 폴더 README의 실행 프롬프트를 그대로
쓰면 됩니다. 공통 관찰 포인트: **계약(스키마·스펙)을 먼저 정하면 병렬화된다**(원리 3),
**QA는 "존재 확인"이 아니라 "경계면 교차 비교"**(원리 4).

- `slide/`: 기준일 셀 하나로 전체가 갱신되는 "살아있는 문서" (`/harness` 오케스트레이터)
- `map/`: `/project-spec-writer`로 스펙 → `/harness`로 팀 구성 (What/Who/How 분리를 직접 설계)
- `dashboard/`: 표준 스키마 확정 후 시뮬레이터·정규화·UI 병렬 개발

### 📙 실무 1 🆕 — `codex-engineering/` 트랙 A (엔지니어링)

- **목표**: Codex를 코드 유지보수 흐름(진단→패치→리뷰→CI)에 넣는다. "초록불 ≠ 정확"을 체험.
- **준비**:
  ```bash
  cd codex-engineering/track-a-engineering
  pip install pytest
  python -m pytest tests/ -q      # → 3 passed (통과하지만…)
  python src/quality_check.py     # → 전 라인 '정상'으로 표시됨 (사실은 위험 수준!)
  ```
- **따라하기** (이 폴더에서 `codex`를 실행하거나 Codex 앱에서 폴더를 열고, 아래를 **입력**):
  ```text
  # 1) 진단 — "고치지 말고 검토만" (Codex는 읽기만 함)
  src/quality_check.py 를 검토해줘. 테스트는 통과하지만 불량률(ppm) 계산과 경계 처리에
  결함이 있는지 근거와 함께 진단해줘. 지금은 수정하지 말고 진단만.

  # 2) 패치 — Codex가 파일 수정·pytest 실행 전에 '승인'을 물음. 확인 후 허용
  방금 진단한 결함 두 가지를 수정하고, tests/에 절대값·경계값 테스트를 추가해
  결함을 실제로 잡도록 강화한 뒤 pytest로 통과를 확인해줘. 무엇을 왜 바꿨는지 요약해줘.

  # 3) 교차 검증 — 관점을 바꿔 다시 검토시키거나, 전용 명령으로
  방금 수정을 리뷰어 관점에서 비판적으로 다시 검토해줘. (또는 터미널에서: codex review --uncommitted)
  ```
- **관찰 포인트**: 테스트가 통과했는데도(준비 단계의 `3 passed`) 진단이 결함을 지적한다 → 그래서 검증은
  테스트와 **별도**로 필요하다(원리 4). 대화형에선 Codex가 **쓰기 전에 승인을 묻는 것**이 안전장치다.
  `codex review`는 diff를 리뷰하므로 **수정을 만든 뒤** 실행하는 게 자연스럽다.
- **자가 점검**
  - [ ] 숨은 결함 2개(ppm 단위 오류, 0 나눗셈)를 스스로/AI로 찾았다
  - [ ] "진단만" 단계에선 Codex가 파일을 안 건드리고, 패치 단계에선 승인을 묻는 것을 확인했다
  - [ ] 패치 후 버그를 일부러 되돌리면 새 테스트가 **빨간불**이 되는지 확인했다
- **자동화(선택)**: 이 흐름을 매 PR마다 반복하려면 `codex exec`·CI로 굳힌다 —
  [트랙 A README §5](codex-engineering/track-a-engineering/README.md) 참고.
- **다음**: 코드를 안 짜는 자동화인 트랙 B로.

### 📙 실무 2 🆕 — `codex-engineering/` 트랙 B (업무 자동화)

- **목표**: 코드 없이 반복 업무(금융 규제 모니터링)를 규칙·형식 계약·자동실행으로 시스템화.
- **준비**: `cd codex-engineering/track-b-ops` 후 `codex --search`(라이브 검색) 실행 또는 Codex 앱.
- **따라하기** (대화형으로 **입력**):
  ```text
  # 1) 조사시키기
  최근 1주 한국 금융 규제·공시 동향을 조사해줘. 자본규제·소비자보호·AML/제재·공시 4개
  영역을 훑고, 각 항목에 출처 기관명과 발표일을 붙여줘. 확인 안 된 건 '[확인 필요]'로.

  # 2) 표로 정리 — 여러 번 하면 분류·영향도 표기가 흔들리는 걸 관찰
  위 내용을 표로 정리해줘. 컬럼: 제목 / 출처 / 발표일 / 분류 / 영향도 / 다음 행동.
  ```
- **관찰 포인트**: 대화형으로 반복하면 형식(분류·영향도 표기)이 매번 흔들린다. 이 흔들림을
  없애려면 형식을 **스키마 계약**으로 고정해야 한다(원리 3) — 그게 아래 자동화다. AGENTS.md의
  "출처 필수·추측 금지"는 프롬프트에 안 써도 지켜진다(원리 2).
- **자동화(형식 보장)**: 매주 똑같은 구조를 보장하려면 `codex exec`의 `--output-schema`를 쓴다.
  ```bash
  codex exec --search --sandbox read-only --output-schema schema/regwatch.schema.json \
    --output-last-message regwatch.json \
    "최근 1주 한국 금융 규제·공시를 조사해 schema에 맞춰 JSON으로 반환. category·impact는 스키마 값만."
  ```
- **자가 점검**
  - [ ] 대화형에서 형식이 흔들리는 문제를 직접 겪었다
  - [ ] 자동화(`--output-schema`) 결과 `regwatch.json`이 스키마 형태(as_of, items[…])·enum을 지킨다
  - [ ] 이 명령을 스케줄러/GitHub Actions에 걸면 정례 자동화가 됨을 이해했다
- **다음**: 세 번째 축인 Tool을 직접 만드는 확장 단계로.

### 📕 확장 🆕 — `tool/` 오픈소스 MCP Tool 만들기

- **목표**: 에이전트가 못 하는 일(정확한 계산)을 결정론적 코드로 위임하는 Tool을 만들고,
  Claude Code·Codex가 **같은 서버를 공유**하게 한다.
- **준비**:
  ```bash
  cd tool/mcp-server
  pip install -r requirements.txt
  python -c "from server import compute_oee; print(compute_oee('B','2026-07-14'))"
  # → 언제나 oee_pct '69.8%' (암산과 달리 값이 고정)
  ```
- **따라하기**:
  ```bash
  # Claude Code에 등록 (register-claude-code.md 참고)
  claude mcp add production-metrics -- python server.py
  #   → Claude Code에서: "production-metrics 도구로 07-14 A·B·C 라인 OEE를 구하고 병목을 정리해줘"

  # 같은 서버를 Codex에도 등록 (register-codex.md 참고, 절대경로 권장)
  codex mcp add production-metrics -- python /절대경로/harness-edu/tool/mcp-server/server.py
  #   → codex 실행 후 입력: "production-metrics MCP로 07-14 B라인 OEE와 불량률을 구하고 개선 우선순위를 제안해줘"
  ```
- **관찰 포인트**: 모델이 OEE를 암산하지 않고 도구를 호출해 정확한 값을 받은 뒤 **해석만**
  얹는다. server.py를 한 줄도 안 바꾸고 두 클라이언트가 공유한다(원리 3 — 한 번 만들어 여러 곳).
- **자가 점검**
  - [ ] Skill(설명서)과 Tool(계산기)의 차이를 설명할 수 있다
  - [ ] `@mcp.tool()` 함수 하나가 곧 하나의 도구임을 이해했다
  - [ ] 같은 서버가 Claude Code·Codex 양쪽에서 동작하는 것을 확인했다
- **심화**: 금융 지표(자기자본비율 등) Tool 만들기, 실제 오픈소스 MCP(FISIS 등) 활용,
  원격(HTTP) MCP로 승격. [`tool/README.md`](tool/README.md) 7절 참고.
